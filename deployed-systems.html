●   Active research · Edge AI · 2026

GW-OS — A 14-agent intelligence pipeline running on local AI.

GW-OS is a fourteen-agent pipeline running on Raspberry Pi hardware with local AI models. Each agent has a single, well-defined job; together they fetch, filter, and write up a daily intelligence brief that lands in the WSL portal — no cloud calls, no external APIs, no data leaving the network.

What the pipeline does

The fourteen agents are organized into three stages: collection, quality control, and publication.

  • Collection — agents pull cybersecurity updates, cyber warnings, world news, economic and financial signals, and AI-industry developments
  • Quality control — agents score, deduplicate, and validate what was collected before anything is written
  • Publication — agents draft the document, format it, and push the result to the WSL portal

Why this matters

Most enterprise monitoring and intelligence work assumes the cloud. Many of the environments that need it most — defense, healthcare, critical infrastructure, schools and non-profits with privacy obligations — cannot or should not send their data there. GW-OS shows that a meaningful, multi-agent intelligence workflow can run entirely on local hardware.

Status

Active development. Pipeline running end-to-end across the fourteen agents. Documentation and a public reference deployment in progress.

What this proves

Multi-agent AI workflows are viable on small, local hardware. The architecture is private by default, air-gap-friendly, and inexpensive enough to deploy at scale. The pattern — fetch, score, write, publish — transfers to many monitoring and intelligence problems beyond the current pipeline.

●   Concluded · Capability proof · 2024–2025

GridWatch Enterprise — Solo capability proof.

GridWatch Enterprise is a unified IPAM (IP address management) and network performance monitoring platform, built solo as a proof that one motivated engineer can ship enterprise-class software with the right tools and discipline.

It is not for sale and is not in active development. The project served its purpose: it proved that the build was possible, that the architecture held up under pressure, and that the lessons could be carried forward into the work that came after — including the engagement model now used with paying clients.

What was built

  • Unified IPAM and network performance monitoring in a single platform
  • Containerized microservices, fourteen Cosign-signed images
  • Air-gap-compatible deployment alongside connected mode
  • Multi-vendor device discovery and monitoring
  • Automated configuration drift checks against DISA STIG baselines
  • Role-based access with structured audit trails

What this proves

Solo-engineered enterprise software is feasible when the architecture is right and the discipline holds. The same approach — security-first, audit-ready, deployed in reversible steps — is now applied on engagements where the deliverable is not a product but a hardened system the client owns afterward.
